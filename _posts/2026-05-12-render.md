首先要回答的问题是何为渲染：

渲染（render）对于浏览器来说，仅代表着一件事，就是将 HTML/CSS 转换为屏幕上像素信息的这一整条完整流水线；对于vue或者react等框架来说，渲染则意味着计算变更后的 UI 应该长什么样（虚拟 DOM），并更新真实 DOM 来匹配这个状态。

```
渲染流水线的大致概念：
HTML 字符串
 → 解析 HTML → DOM 树
 → 解析 CSS → CSSOM 树
 → DOM + CSSOM → Render 树
 → 布局（Layout）：计算每个元素的位置和大小
 → 绘制（Paint）：生成绘制指令
 → 合成（Composite）：拆层、GPU 合成、输出到屏幕
```

而渲染发生的时机，就与事件循环有密切的关联。

对于“输入地址按下回车键后，发生了什么事”这一经典面试题，可以说整个过程主要分为两个阶段，一是网络（拿到html字符串），二就是渲染（解析html）。网络的过程也特别复杂，但这里主要专注于渲染。

网络进程中的网络线程负责拿到html字符串，并生成一个渲染任务交给渲染主线程的消息队列。通过事件循环机制，渲染主线程会提取渲染任务，然后进行渲染流程。

更偏向实现细节的渲染流水线可以这样细分，Chrome DevTools的Performance面板基本按照这个流程展示：

![image-20260513175609968](/assets/blog_res/2026-05-12-render.assets/image-20260513175609968.png)

渲染的第一步是解析HTML字符串（parse HTML），这个过程会产生DOM树和CSSOM树，树中每个节点都是一个对象，用typeof可以确认这一点，如果想要更进一步查询是什么类型的节点，可以使用instanceof。

```js
const div = document.querySelector('div');
const textNode = div.firstChild; // 文本节点

typeof div; // "object"
typeof textNode; // "object" ← 所有节点都一样

// 进一步判断节点类型
div.nodeType; // 1（元素节点）
textNode.nodeType; // 3（文本节点）

// 或
div instanceof HTMLElement; // true
textNode instanceof Text; // true
```

![image-20260513181333397](/assets/blog_res/2026-05-12-render.assets/image-20260513181333397.png)

严格来说，这里底层形成的dom树是c++的对象结构，而在js操作使用的是js的对象结构，两者并不是完全等同，实际上js操作时会在c++的对象结构上包装一层js（这个包装由绑定层完成，在chrome里是blink引擎执行，V8本身只管执行 JS，它不认识 DOM，`document` 这个对象是 Blink 注入给 V8 的），使得js能够操作。

这一步的关键意义是让后续的步骤不用再直接操作字符串，而是能够操作结构更清晰的dom树。

![image-20260513182454581](/assets/blog_res/2026-05-12-render.assets/image-20260513182454581.png)

cssom的树形结构：

根节点代表着网页中所有的样式表（\<style>内部样式表，\<link ...>外部样式表，浏览器默认样式表等等，\<div style="">内联样式严格来说不属于 `document.styleSheets`，不存在于 CSSOM 的树结构中。它是 DOM 元素上的 `element.style` 属性，在样式计算（style recalculation）这一步和 CSSOM 规则一起参与层叠，算出最终值），根节点会下边对应产生相同数量的节点。

> 样式表按来源可以分为三种：
>
> | 类型               | 来源                                                       | 在 CSSOM 中的位置                                 |
> | :----------------- | :--------------------------------------------------------- | :------------------------------------------------ |
> | **用户代理样式表** | 浏览器自带（比如 `<h1>` 默认加粗、`<a>` 默认蓝色加下划线） | `document.styleSheets` 不暴露，但对开发者工具可见 |
> | **用户样式表**     | 用户自定义（很少用，浏览器扩展也算）                       | 普通 JS 无法访问                                  |
> | **作者样式表**     | 网页开发者写的                                             | `document.styleSheets` 可遍历                     |
>
> 作者样式表又分三种形式：
>
> ```html
> <!-- 1. 外部样式表 -->
> <link rel="stylesheet" href="style.css">
> 
> <!-- 2. 内部样式表 -->
> <style>
>   body { margin: 0; }
> </style>
> 
> <!-- 3. 内联样式 -->
> <div style="color: red;">hello</div>
> ```
>
> 但内联样式不在 CSSOM 树里，它直接挂在 DOM 节点的 `element.style` 上，不参与样式表的规则层叠，只参与最终计算。
>
> 所以 CSSOM 树实际管理的是外部样式表和内部样式表。用户代理样式表的具体样式可以在blink源码中查看。

js不仅可以操作dom，也可以操作cssom，即外部样式表和内部样式表，内联样式通过dom树操作，最后也参与在cssom的更新。
