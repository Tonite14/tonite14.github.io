首先要回答的问题是何为渲染：

渲染（render）对于浏览器来说，仅代表着一件事，就是将 HTML/CSS 转换为屏幕上像素信息的这一整条完整流水线；对于vue或者react等框架来说，渲染则意味着计算变更后的 UI 应该长什么样（虚拟 DOM），并更新真实 DOM 来匹配这个状态。

```
渲染流水线的大致概念：
HTML 字符串
 → 解析 HTML → DOM 树
 → 解析 CSS → CSSOM 树
 → DOM + CSSOM → Render 树
 → 布局（Layout）：计算每个元素的位置和大小
 → 绘制（Paint）：生成绘制指令
 → 合成（Composite）：拆层、GPU 合成、输出到屏幕
```

而渲染发生的时机，就与事件循环有密切的关联。

对于“输入地址按下回车键后，发生了什么事”这一经典面试题，可以说整个过程主要分为两个阶段，一是网络（拿到html字符串），二就是渲染（解析html）。网络的过程也特别复杂，但这里主要专注于渲染。

网络进程中的网络线程负责拿到html字符串，并生成一个渲染任务交给渲染主线程的消息队列。通过事件循环机制，渲染主线程会提取渲染任务，然后进行渲染流程。

更偏向实现细节的渲染流水线可以这样细分，Chrome DevTools的Performance面板基本按照这个流程展示：

## 解析HTML：

渲染的第一步是解析HTML字符串（字节流）（parse HTML），这个过程会产生DOM树和CSSOM树，树中每个节点都是一个对象，用typeof可以确认这一点，如果想要更进一步查询是什么类型的节点，可以使用instanceof。

```js
const div = document.querySelector('div');
const textNode = div.firstChild; // 文本节点

typeof div; // "object"
typeof textNode; // "object" ← 所有节点都一样

// 进一步判断节点类型
div.nodeType; // 1（元素节点）
textNode.nodeType; // 3（文本节点）

// 或
div instanceof HTMLElement; // true
textNode instanceof Text; // true
```

![image-20260513181333397](/assets/blog_res/2026-05-12-render.assets/image-20260513181333397.png)

严格来说，这里底层形成的dom树是c++的对象结构，而在js操作使用的是js的对象结构，两者并不是完全等同，实际上js操作时会在c++的对象结构上包装一层js（这个包装由绑定层完成，在chrome里是blink引擎执行，V8本身只管执行 JS，它不认识 DOM，`document` 这个对象是 Blink 注入给 V8 的），使得js能够操作。

这一步的关键意义是让后续的步骤不用再直接操作字符串，而是能够操作结构更清晰的dom树。

![image-20260513182454581](/assets/blog_res/2026-05-12-render.assets/image-20260513182454581.png)

cssom的树形结构：

cssom也是对象结构，根节点代表着网页中所有的样式表（\<style>内部样式表，\<link ...>外部样式表，浏览器默认样式表等等，\<div style="">内联样式严格来说不属于 `document.styleSheets`，不存在于 CSSOM 的树结构中。它是 DOM 元素上的 `element.style` 属性，在样式计算（style recalculation）这一步和 CSSOM 规则一起参与层叠，算出最终值），根节点会下边对应产生相同数量的节点。

> 样式表按来源可以分为三种：
>
> | 类型               | 来源                                                       | 在 CSSOM 中的位置                                 |
> | :----------------- | :--------------------------------------------------------- | :------------------------------------------------ |
> | **用户代理样式表** | 浏览器自带（比如 `<h1>` 默认加粗、`<a>` 默认蓝色加下划线） | `document.styleSheets` 不暴露，但对开发者工具可见 |
> | **用户样式表**     | 用户自定义（很少用，浏览器扩展也算）                       | 普通 JS 无法访问                                  |
> | **作者样式表**     | 网页开发者写的                                             | `document.styleSheets` 可遍历                     |
>
> 作者样式表又分三种形式：
>
> ```html
> <!-- 1. 外部样式表 -->
> <link rel="stylesheet" href="style.css">
> 
> <!-- 2. 内部样式表 -->
> <style>
>   body { margin: 0; }
> </style>
> 
> <!-- 3. 内联样式 -->
> <div style="color: red;">hello</div>
> ```
>
> 但内联样式不在 CSSOM 树里，它直接挂在 DOM 节点的 `element.style` 上，不参与样式表的规则层叠，只参与最终计算。
>
> 所以 CSSOM 树实际管理的是外部样式表和内部样式表。用户代理样式表的具体样式可以在blink源码中查看。

js不仅可以操作dom，也可以操作cssom，即外部样式表和内部样式表，内联样式通过dom树操作，最后也参与在cssom的更新。

![image-20260513182426915](/assets/blog_res/2026-05-12-render.assets/image-20260513182426915.png)

已知HTML字节流是包含一般标签的一般内容和css代码的，也已知parse阶段会生成dom和cssom，所以对于一般的dom节点和带有css内容的部分的处理肯定是具有不同的。那么在HTML解析过程中遇到css代码会怎么办？

为了提高解析效率，浏览器会启动一个预解析器（这里的解析与主线程的解析含义不同，可以理解为下载）率先下载和解析css。而渲染主线程在遇到css相关内容时，因为已知预解析线程在处理相关数据，会跳过css的相关内容，继续往后解析，直到预解析线程完成解析，就合并入cssom。

因为有了不同线程（这些不同线程由浏览器负责）的参与，css代码不会阻塞html的解析。

![image-20260514005742631](/assets/blog_res/2026-05-12-render.assets/image-20260514005742631.png)

在谈论完dom和cssom后，HTML字节流中还有一个重要内容，就是JS代码，JS代码并不和dom与cssom一样，可能后续需要多次更新，JS代码只执行完毕一遍即可。

又因为JS代码有可能修改当前已有的dom，dom树是一边解析一边动态生成的，所以dom树最高效的修改方式当然是让js先执行完再确立最后的dom，不必在渲染完成后再根据js修改dom造成返工。因此在解析过程中如果遇到JS代码，渲染主线程必须暂停一切行为，等待下载完毕，并启动v8执行完js后再继续。

在这个过程中预解析线程会分担一部分的下载负担（例如引用外部js时）。

![image-20260514013520784](/assets/blog_res/2026-05-12-render.assets/image-20260514013520784.png)

## 样式计算：

经过HTML Parser后，得到了dom树和cssom树，下一步就是根据两棵树进行样式计算（Recalculate Style）。经过样式计算后，每个节点就具有了最终的完整的，即计算后的样式（Computed Style）。

虽然我们设置的样式值往往不超过十个，但此时每一个样式都会具有自己的值，这个最终值的决定就涉及了css属性值的计算过程（css两块最重要的知识就是css属性的计算过程和视觉格式化模型）。

完成之后，会得到一棵带有样式的dom树。

![image-20260514014405686](/assets/blog_res/2026-05-12-render.assets/image-20260514014405686.png)

## 布局：

保证每个节点都具有全部的css样式之后，就会进入布局阶段。这一阶段会根据每个节点的样式来算出它们各自的尺寸和位置。布局计算规则可能非常复杂，主要涉及视觉格式化相关的知识（包含块等）。

这棵布局树并不是dom树，其中的节点也不是dom节点对象，而是c++对象，js是无法获取的。

![image-20260514015606370](/assets/blog_res/2026-05-12-render.assets/image-20260514015606370.png)

布局得到的树与dom树不一定是一一对应的，有些隐藏的元素（例如head、link和meta等标签，chromium源码中的默认样式表设置是display: none）不具备几何信息，即尺寸和位置，不会出现在布局树中。

![image-20260514020527973](/assets/blog_res/2026-05-12-render.assets/image-20260514020527973.png)

布局树的元素也可能相对dom树增加，例如使用::before时，因为引入了新的几何信息。

![image-20260514020536574](/assets/blog_res/2026-05-12-render.assets/image-20260514020536574.png)

这里的节点差异涉及了两个规则：内容必须在行盒中，行盒不能和块盒相邻。根据内容必须在行盒中的要求，在文本a和c之前会生成匿名行盒。

![image-20260514020952529](/assets/blog_res/2026-05-12-render.assets/image-20260514020952529.png)

## 分层：

分层的目的是降低每次页面更新时的更新量，可以理解成每次渲染更新时只更新页面上需要更新的层，而其他层无需改变，从而降低开销。在以前的浏览器的没有这个功能的，现代浏览器才具备。

![image-20260514143057415](/assets/blog_res/2026-05-13-render.assets/image-20260514143057415.png)

这一层的相关信息可以在开发者工具查看（layers工具，中文名为图层）。层次不会划分得特别多，每个浏览器的划分策略都不同，可能非常复杂。

跟堆叠上下文有关的属性（z-index、opacity等）可能会影响到分层结果。开发者无法直接控制层的形成，但可以通过嵌入css等方式来提示浏览器分层（will-change），形象的理解可以类比于告诉浏览器这个元素我会频繁更新，因此浏览器为便于更新元素，可能会对这个元素进行单独分层。但对于这个策略不能滥用，因为我们常常无法保证自己设置的策略的效率会优于浏览器的默认策略，而我们采用分层的目的也恰恰只是为了提高效率。一般只在排查问题时发现渲染速度慢就是因为浏览器的分层策略不足时，我们才会引导浏览器进行分层。

## 绘制：

绘制是基于已有的分层，为每一层生成如何绘制的指令（这里的绘制指令与canvas基本相似，canvas调用的正是浏览器内核的绘制功能）。

![image-20260514143417276](/assets/blog_res/2026-05-13-render.assets/image-20260514143417276.png)

完成绘制后，渲染主线程的工作就到此为止，剩余步骤交给其他线程完成。这种优化设计极大提升了渲染整体效率。

![image-20260514143513155](/assets/blog_res/2026-05-13-render.assets/image-20260514143513155.png)
