番外 浏览器渲染原理

发表于 2026/05/13 更新于 2026/07/19

首先要回答的问题是何为渲染：

渲染（render）对于浏览器来说，仅代表着一件事，就是将 HTML/CSS 转换为屏幕上像素信息的这一整条完整流水线；对于vue或者react等框架来说，渲染则意味着计算变更后的 UI 应该长什么样（虚拟 DOM），并更新真实 DOM 来匹配这个状态。

渲染流水线的大致概念：
HTML 字符串
 → 解析 HTML → DOM 树
 → 解析 CSS → CSSOM 树
 → DOM + CSSOM → Render 树
 → 布局（Layout）：计算每个元素的位置和大小
 → 绘制（Paint）：生成绘制指令
 → 合成（Composite）：拆层、GPU 合成、输出到屏幕

而渲染发生的时机，就与事件循环有密切的关联。

对于“输入地址按下回车键后，发生了什么事”这一经典面试题，可以说整个过程主要分为两个阶段，一是网络（拿到html字符串），二就是渲染（解析html）。网络的过程也特别复杂，但这里主要专注于渲染。

网络进程中的网络线程负责拿到html字符串，并生成一个渲染任务交给渲染主线程的消息队列。通过事件循环机制，渲染主线程会提取渲染任务，然后进行渲染流程。

更偏向实现细节的渲染流水线可以这样细分，Chrome DevTools的Performance面板基本按照这个流程展示：

解析HTML：

渲染的第一步是解析HTML字符串（字节流）（parse HTML），这个过程会产生DOM树和CSSOM树，树中每个节点都是一个对象，用typeof可以确认这一点，如果想要更进一步查询是什么类型的节点，可以使用instanceof。

严格来说，这里底层形成的dom树是c++的对象结构，而在js操作使用的是js的对象结构，两者并不是完全等同，实际上js操作时会在c++的对象结构上包装一层js（这个包装由绑定层完成，在chrome里是blink引擎执行，V8本身只管执行 JS，它不认识 DOM，document 这个对象是 Blink 注入给 V8 的），使得js能够操作。

这一步的关键意义是让后续的步骤不用再直接操作字符串，而是能够操作结构更清晰的dom树。

cssom的树形结构：

cssom也是对象结构，根节点代表着网页中所有的样式表（<style>内部样式表，<link …>外部样式表，浏览器默认样式表等等，<div style=”“>内联样式严格来说不属于 document.styleSheets，不存在于 CSSOM 的树结构中。它是 DOM 元素上的 element.style 属性，在样式计算（style recalculation）这一步和 CSSOM 规则一起参与层叠，算出最终值），根节点会下边对应产生相同数量的节点。

样式表按来源可以分为三种：

类型来源在 CSSOM 中的位置
用户代理样式表浏览器自带（比如 <h1> 默认加粗、<a> 默认蓝色加下划线）document.styleSheets 不暴露，但对开发者工具可见
用户样式表用户自定义（很少用，浏览器扩展也算）普通 JS 无法访问
作者样式表网页开发者写的document.styleSheets 可遍历

但内联样式不在 CSSOM 树里，它直接挂在 DOM 节点的 element.style 上，不参与样式表的规则层叠，只参与最终计算。

所以 CSSOM 树实际管理的是外部样式表和内部样式表。用户代理样式表的具体样式可以在blink源码中查看。

js不仅可以操作dom，也可以操作cssom，即外部样式表和内部样式表，内联样式通过dom树操作，最后也参与在cssom的更新。

已知HTML字节流是包含一般标签的一般内容和css代码的，也已知parse阶段会生成dom和cssom，所以对于一般的dom节点和带有css内容的部分的处理肯定是具有不同的。那么在HTML解析过程中遇到css代码会怎么办？

为了提高解析效率，浏览器会启动一个预解析器（这里的解析与主线程的解析含义不同，可以理解为下载）率先下载和解析css。而渲染主线程在遇到css相关内容时，因为已知预解析线程在处理相关数据，会跳过css的相关内容，继续往后解析，直到预解析线程完成解析，就合并入cssom。

因为有了不同线程（这些不同线程由浏览器负责）的参与，css代码不会阻塞html的解析。

在谈论完dom和cssom后，HTML字节流中还有一个重要内容，就是JS代码，JS代码并不和dom与cssom一样，可能后续需要多次更新，JS代码只执行完毕一遍即可。

又因为JS代码有可能修改当前已有的dom，dom树是一边解析一边动态生成的，所以dom树最高效的修改方式当然是让js先执行完再确立最后的dom，不必在渲染完成后再根据js修改dom造成返工。因此在解析过程中如果遇到JS代码，渲染主线程必须暂停一切行为，等待下载完毕，并启动v8执行完js后再继续。

在这个过程中预解析线程会分担一部分的下载负担（例如引用外部js时）。

样式计算：

经过HTML Parser后，得到了dom树和cssom树，下一步就是根据两棵树进行样式计算（Recalculate Style）。经过样式计算后，每个节点就具有了最终的完整的，即计算后的样式（Computed Style）。

虽然我们设置的样式值往往不超过十个，但此时每一个样式都会具有自己的值，这个最终值的决定就涉及了css属性值的计算过程（css两块最重要的知识就是css属性的计算过程和视觉格式化模型）。

完成之后，会得到一棵带有样式的dom树。

布局：

保证每个节点都具有全部的css样式之后，就会进入布局阶段。这一阶段会根据每个节点的样式来算出它们各自的尺寸和位置。布局计算规则可能非常复杂，主要涉及视觉格式化相关的知识（包含块等）。

这棵布局树并不是dom树，其中的节点也不是dom节点对象，而是c++对象，js是无法获取的。

布局得到的树与dom树不一定是一一对应的，有些隐藏的元素（例如head、link和meta等标签，chromium源码中的默认样式表设置是display: none）不具备几何信息，即尺寸和位置，不会出现在布局树中。

布局树的元素也可能相对dom树增加，例如使用::before时，因为引入了新的几何信息。

这里的节点差异涉及了两个规则：内容必须在行盒中，行盒不能和块盒相邻。根据内容必须在行盒中的要求，在文本a和c之前会生成匿名行盒。

分层：

分层的目的是降低每次页面更新时的更新量，可以理解成每次渲染更新时只更新页面上需要更新的层，而其他层无需改变，从而降低开销。在以前的浏览器的没有这个功能的，现代浏览器才具备。

这一层的相关信息可以在开发者工具查看（layers工具，中文名为图层）。层次不会划分得特别多，每个浏览器的划分策略都不同，可能非常复杂。

跟堆叠上下文有关的属性（z-index、opacity等）可能会影响到分层结果。开发者无法直接控制层的形成，但可以通过嵌入css等方式来提示浏览器分层（will-change），形象的理解可以类比于告诉浏览器这个元素我会频繁更新，因此浏览器为便于更新元素，可能会对这个元素进行单独分层。但对于这个策略不能滥用，因为我们常常无法保证自己设置的策略的效率会优于浏览器的默认策略，而我们采用分层的目的也恰恰只是为了提高效率。一般只在排查问题时发现渲染速度慢就是因为浏览器的分层策略不足时，我们才会引导浏览器进行分层。

绘制：

绘制是基于已有的分层，为每一层生成如何绘制的指令集（这里的绘制指令与canvas基本相似，canvas调用的正是浏览器内核的绘制功能）。

完成绘制后，渲染主线程的工作就到此为止，剩余步骤交给其他线程完成。这种优化设计极大提升了渲染整体效率。

分块：

分块（tiling）会将每一层分为多个小的区域，可以用于区分每一层中的渲染优先级，例如会优先将靠近视口的区域绘出。

分块的工作是交给多个线程同时进行的。合成线程也在渲染进程当中，并且合成线程会从线程池取出多个线程来完成工作。

光栅化：

光栅化（raster）是将每个块变为位图，这个过程会优先处理靠近视口的块。

浏览器会调用gpu进程来完成gpu加速。

画图：

渲染主线程和合成线程都属于渲染进程，渲染进程在一个沙盒中，与操作系统硬件隔离，不受外界干扰。这样的好处是在网页浏览过程中遭受的攻击不会直接影响操作系统，但因此合成线程无法直接联系操作系统硬件，需要通过GPU线程的系统调用才能将quad交给硬件。

最终完整过程：

容易发现，在渲染主线程交付完自己的五个任务给合成线程后（完成他们之间的同步关系），如果有什么操作是只在渲染主线程或者合成线程上进行，是不会影响另一边的线程的。

例如滚动滚动条时，只会影响视口位置，这些操作都可以在合成线程完成，因此即使渲染主线程在忙，也可以正常滚动滚动条。

常见面试题：

浏览器是如何渲染页面的？

当浏览器的网络线程收到 HTML 文档后，会产生一个渲染任务，并将其传递给渲染主线程的消息队列。

在事件循环机制的作用下，渲染主线程取出消息队列中的渲染任务，开启渲染流程。

整个渲染流程分为多个阶段，分别是： HTML 解析、样式计算、布局、分层、绘制、分块、光栅化、画

每个阶段都有明确的输入输出，上一个阶段的输出会成为下一个阶段的输入。

这样，整个渲染流程就形成了一套组织严密的生产流水线。

渲染的第一步是解析 HTML。

解析过程中遇到 CSS 解析 CSS，遇到 JS 执行 JS。为了提高解析效率，浏览器在开始解析前，会启动一个预解析的线程，率先下载 HTML 中的外部 CSS 文件和 外部的 JS 文件。

如果主线程解析到link位置，此时外部的 CSS 文件还没有下载解析好，主线程不会等待，继续解析后续的 HTML。这是因为下载和解析 CSS 的工作是在预解析线程中进行的。这就是 CSS 不会阻塞 HTML 解析的根本原因。

如果主线程解析到script位置，会停止解析 HTML，转而等待 JS 文件下载好，并将全局代码解析执行完成后，才能继续解析 HTML。这是因为 JS 代码的执行过程可能会修改当前的 DOM 树，所以 DOM 树的生成必须暂停。这就是 JS 会阻塞 HTML 解析的根本原因。

第一步完成后，会得到 DOM 树和 CSSOM 树，浏览器的默认样式、内部样式、外部样式、行内样式均会包含在 CSSOM 树中。

渲染的下一步是样式计算。

主线程会遍历得到的 DOM 树，依次为树中的每个节点计算出它最终的样式，称之为 Computed Style。

在这一过程中，很多预设值会变成绝对值，比如red会变成rgb(255,0,0)；相对单位会变成绝对单位，比如em会变成px

这一步完成后，会得到一棵带有样式的 DOM 树。

接下来是布局，布局完成后会得到布局树。

布局阶段会依次遍历 DOM 树的每一个节点，计算每个节点的几何信息。例如节点的宽高、相对包含块的位置。

大部分时候，DOM 树和布局树并非一一对应。

比如display:none的节点没有几何信息，因此不会生成到布局树；又比如使用了伪元素选择器，虽然 DOM 树中不存在这些伪元素节点，但它们拥有几何信息，所以会生成到布局树中。还有匿名行盒、匿名块盒等等都会导致 DOM 树和布局树无法一一对应。

从这个角度也可以看到，HTML、CSS、JS 三者对 DOM 的作用是不同层面的：HTML 在解析阶段定义 DOM 的初始结构；CSS 不修改 DOM 节点本身，但通过 display: none、伪元素、visibility 等决定节点在渲染管线中的呈现方式（操作的是布局树和渲染层）；JS 在运行时通过 DOM API 增删改节点，是唯一能在页面加载完成后继续修改 DOM 树的方式。三者的分工可以概括为——HTML 定义 DOM，JS 动态改 DOM，CSS 决定 DOM 怎么呈现。

下一步是分层

主线程会使用一套复杂的策略对整个布局树中进行分层。

分层的好处在于，将来某一个层改变后，仅会对该层进行后续处理，从而提升效率。

滚动条、堆叠上下文、transform、opacity 等样式都会或多或少的影响分层结果，也可以通过will-change属性更大程度的影响分层结果。

再下一步是绘制

主线程会为每个层单独产生绘制指令集，用于描述这一层的内容该如何画出来。

完成绘制后，主线程将每个图层的绘制信息提交给合成线程，剩余工作将由合成线程完成。

合成线程首先对每个图层进行分块，将其划分为更多的小区域。

它会从线程池中拿取多个线程来完成分块工作。

分块完成后，进入光栅化阶段。

合成线程会将块信息交给 GPU 进程，以极高的速度完成光栅化。

GPU 进程会开启多个线程来完成光栅化，并且优先处理靠近视口区域的块。

光栅化的结果，就是一块一块的位图

最后一个阶段就是画了

合成线程拿到每个层、每个块的位图后，生成一个个「指引（quad）」信息。

指引会标识出每个位图应该画到屏幕的哪个位置，以及会考虑到旋转、缩放等变形。

变形发生在合成线程，与渲染主线程无关，这就是transform效率高的本质原因。

合成线程会把 quad 提交给 GPU 进程，由 GPU 进程产生系统调用，提交给 GPU 硬件，完成最终的屏幕成像。

JS 执行和渲染为什么是分开的？

DOM 操作和 DOM 渲染是浏览器事件循环中的两个独立阶段。JS 在宏任务中随便怎么改 DOM，全是改内存里的 DOM 树，浏览器不会立刻渲染：

事件循环：
├── 宏任务：执行 JS（随便怎么改 DOM，全在内存里）
│     i=0  div.textContent = 1    ← 只改了内存里的 DOM 树
│     i=1  div.textContent = 2    ← 覆盖上一次的值
│     ...
│     i=99 div.textContent = 100  ← 最终值
│
├── 微任务队列：排空
│
└── 渲染阶段（不一定每轮都触发）：
      layout → paint → composite    ← 这时候才看到 "100"

所以循环改 100 次 DOM，浏览器一次都没有渲染，只是改了 100 次内存里的 DOM 树。等 JS 执行完毕、微任务清空后，浏览器在渲染阶段拿到最新的 DOM 树状态，layout + paint 一次出结果。

之前提到的强制回流就是这个机制的例外：如果在改 DOM 之后立即读取 offsetHeight，浏览器为了给出准确的布局值，会被迫在当前 JS 执行中插入一次同步 layout——破坏了”改完再渲染”的批处理优化。

什么是 reflow？

reflow 的本质就是重新计算 layout 树。

当进行了会影响布局树的操作后，需要重新计算布局树，会引发 layout。

为了避免连续的多次操作导致布局树反复计算，浏览器会合并这些操作（浏览器的优化，将这些操作推入同一队列），当 JS 代码全部完成后再进行统一计算。所以，改动属性造成的 reflow 是异步完成的。

也同样因为如此，当 JS 获取布局属性时（同步任务），就可能造成无法获取到最新的布局信息（仍在异步队列中）。

浏览器在反复权衡下，为保证一定能得到最新数据，最终决定在设置属性时不会立即 reflow，但获取属性时立即 reflow。

什么是 repaint？

repaint 的本质就是重新根据分层信息计算了绘制指令。

当改动了可见样式后，就需要重新计算，会引发 repaint。

由于元素的布局信息也属于可见样式，所以 reflow 一定会引起 repaint。

为什么 transform 的效率高？

因为 transform 既不会影响布局也不会影响绘制指令，它影响的只是渲染流程的最后一个「draw」阶段。

由于 draw 阶段在合成线程中，所以 transform 的变化几乎不会影响渲染主线程。反之，渲染主线程无论如何忙碌，也不会影响 transform 的变化。

如果使用animation，甚至不需要修改cssom树，因此完全不会影响主线程；反过来说，主线程在执行的任务也不会影响这里transform的执行。
